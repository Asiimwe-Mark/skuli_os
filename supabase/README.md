# Supabase

This directory holds everything database-related: schema, migrations, edge
functions, and the consolidated schema for fresh setups.

## Layout

```
supabase/
├── schema.sql                  # Canonical end-state schema (224 KB, generated)
├── migrations/                 # 29 canonical migrations, run by `supabase db push`
│   ├── 0001_extensions.sql
│   ├── 0002_enums.sql
│   ├── 0003_helper_functions.sql
│   ├── 0004..0012_*.sql        # Tables grouped by concern
│   ├── 0013_indexes.sql
│   ├── 0014..0017_*.sql        # RLS enable + policies
│   ├── 0018_rls_hardening.sql
│   ├── 0019..0022_*.sql        # Functions grouped by concern
│   ├── 0023_triggers.sql
│   ├── 0024_views.sql
│   ├── 0025_storage_buckets.sql
│   ├── 0026_grants.sql
│   ├── 0027_seeds.sql
│   ├── 0028_column_drops.sql   # Idempotent column drops for legacy DBs
│   └── 0029_finalize.sql
├── functions/                  # Edge functions (Deno)
└── README.md                   # You are here
```

## Numbering scheme

Files are 4-digit zero-padded so the order is obvious (`0001` runs before `0100`).
Numbering does NOT track historical migration order — it tracks dependency order.
A clean stream means "apply 0001, then 0002, ..." and the result is a working DB.

If you need to add a new migration, the next number is `0030_*.sql`. The
canonical end-state SQL is regenerated by `node scripts/build-schema.mjs`.

## When to use what

| Scenario | Use this |
|---|---|
| Live database upgrade | `supabase db push` — runs the 29 migrations in order |
| Fresh Supabase project setup | `supabase/schema.sql` in the SQL editor (faster) OR `supabase db push` |
| Disaster recovery (lost history) | `supabase/schema.sql` in the SQL editor |
| Adding a feature requiring a schema change | Add `<NNNN>_name.sql` to `migrations/`, regenerate `schema.sql`, commit both |
| Wiping a remote DB and starting over | See `docs/RESET_DATABASE.md` |

## schema.sql

`schema.sql` is the canonical end-state of the database, generated by
concatenating all 29 migration files in numeric order. It exists for two
use cases:

1. **Fresh project setup** — running `schema.sql` once is faster and more
   reliable than `supabase db push` against 29 individual files.
2. **Disaster recovery** — if the migration history is ever lost or
   corrupted, `schema.sql` is the single source of truth for the schema.

It is **not** a replacement for the migrations on a live database. The
`CREATE OR REPLACE` and `IF NOT EXISTS` guards make most statements
idempotent, but `CREATE POLICY` (without IF NOT EXISTS) and
`ALTER TABLE ... ENABLE ROW LEVEL SECURITY` will fail on a partially-
migrated database. Use `supabase db push` for live updates.

To regenerate `schema.sql` after changing any migration:

```bash
node scripts/build-schema.mjs
```

## Storage buckets

The four buckets the application uses are created in `0025_storage_buckets.sql`:

- `staff-photos` — public read
- `student-photos` — public read
- `school-assets` — public read
- `report-cards` — private (signed URLs only)

Each bucket has matching RLS policies on `storage.objects` in the same
file. Adding a new bucket requires a new migration.

## Functions

Edge functions live in `functions/`. Deploy with:

```bash
supabase functions deploy <function-name>
```

## Gate tests

`tests/schema-consistency.test.ts` enforces the contract between the
migrations and the application code:

- Every `.from('table')` call must resolve to a `CREATE TABLE` or
  `CREATE VIEW` in some migration.
- Every `.rpc('fn')` call must resolve to a `CREATE FUNCTION` in some
  migration.
- Every `.storage.from('bucket')` call must resolve to an
  `INSERT INTO storage.buckets` in some migration.
- Every table the migrations create must be referenced by app code or
  marked `-- INTENTIONALLY-UNUSED` in a comment block.

This is the proof the schema can't drift from the app. Run with
`npx vitest run tests/schema-consistency.test.ts`.
